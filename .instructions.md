# Pet E-commerce API - Code Construction Guide

This document defines the coding standards and architectural patterns for the Pet E-commerce API project. AI assistants should follow these guidelines when generating or modifying code.

## Project Overview

- **Framework**: Express.js with TypeScript
- **Database**: MongoDB with TypeORM
- **Architecture**: Layered architecture with Routes → Services → Repositories pattern
- **Target**: ES2020 with strict TypeScript compilation

## Project Structure

```
src/
├── index.ts                    # Application entry point
├── app/
│   ├── config/                # Configuration files
│   ├── database/              # Database initialization and setup

```

## Coding Standards

### TypeScript Configuration

- **Target**: ES2020
- **Module System**: CommonJS
- **Strict Mode**: Enabled (all strict options active)
- **Path Aliases**: Use `@/*` for imports from `src/`
- **Decorators**: Enabled (for TypeORM)

### Import Statements

Always use relative imports from the `src/` directory root using path aliases:

```typescript
// ✓ Good
import { AppDataSource } from '@/app/database';
import { Category } from '@/app/entities/Categories';
import { NotFoundError } from '@/app/exceptions/AppError';

// ✗ Avoid
import { AppDataSource } from '../../../app/database';
```

## Feature Module Pattern

Each feature should follow this structure:

```
features/{featureType}/{featureName}/
├── {FeatureName}Service.ts    # Business logic
├── {FeatureName}Route.ts      # Route definitions
└── index.ts                   # Barrel export
```

### Naming Conventions

- **Classes**: PascalCase (`CategoryService`, `CustomerRoute`)
- **Files**: Match class name exactly (`CategoryService.ts`)
- **Methods**: camelCase (`getAllCategories()`, `getSubcategoriesById()`)
- **Variables**: camelCase (`categoriesData`, `customerId`)
- **Constants**: UPPER_SNAKE_CASE (`API_VERSION`, `CACHE_TTL`)
- **Database Collections**: lowercase with underscores (`categories`, `customers`, `orders`)

## Service Class Pattern

Services handle all business logic and database operations. Follow this template:

```typescript
import { AppDataSource } from '@/app/database';
import { EntityName } from '@/app/entities/EntityName';
import { NotFoundError, BadRequestError } from '@/app/exceptions/AppError';

export class {EntityName}Service {
  private repo = AppDataSource.getMongoRepository({EntityName});

  /**
   * Brief description of what the method does
   */
  async {methodName}(params): Promise<ReturnType> {
    // Business logic here
    // Use custom exceptions for errors
    throw new NotFoundError('Resource not found');
  }
}
```

### Service Guidelines

- Each service is a class that wraps a single entity's repository
- All methods should be async (use `async/await`)
- Use JSDoc comments for public methods
- Validate data and throw appropriate custom errors
- Handle MongoDB ObjectId validation using `ObjectId.isValid()`
- Return partial/selected fields when appropriate to avoid exposing sensitive data

## Route Handler Pattern

Routes define HTTP endpoints and delegate to services. Follow this template:

```typescript
import { Router, Request, Response, NextFunction } from 'express';
import { {ServiceName} } from './{ServiceName}';

const router = Router();
const {serviceName} = new {ServiceName}();

router.get('/endpoint/:id', async (req: Request, res: Response, next: NextFunction) => {
  try {
    const result = await {serviceName}.method(req.params.id);
    res.json({
      success: true,
      data: result
    });
  } catch (error) {
    next(error);
  }
});

export default router;
```

### Route Guidelines

- Routes should be minimal - delegate logic to services
- Always wrap service calls in try-catch and pass errors to `next(error)`
- Use consistent response format: `{ success: boolean, data: any }`
- Validate request parameters before passing to service
- Use `_req` prefix for unused request object: `(_req: Request, ...)`
- Return proper HTTP status codes (200, 201, 400, 404, 409, etc.)

## Entity (Model) Definition

Entities use TypeORM decorators for MongoDB. Follow this pattern:

```typescript
import { Entity, ObjectIdColumn, Column, CreateDateColumn } from 'typeorm';
import { ObjectId } from 'mongodb';

@Entity('collection_name')
export class EntityName {
  @ObjectIdColumn()
  _id: ObjectId;

  @Column()
  fieldName: string;

  @Column({ unique: true })
  uniqueField: string;

  @Column({ default: true })
  is_active: boolean;

  @CreateDateColumn({ name: 'created_at' })
  created_at: Date;

  // Optional: UpdateDateColumn for updated_at
  // Optional: Embedded objects or subdocuments
}
```

### Entity Guidelines

- Use `@ObjectIdColumn()` for MongoDB `_id` field
- Use snake_case for column names in database
- Use `is_active` as boolean flag for soft deletes
- Use `created_at` for creation timestamp with `@CreateDateColumn()`
- Use `@Column(() => SubEntity)` for embedded objects
- Keep entities focused - one entity per file

## Error Handling

Always use custom `AppError` classes from `@/app/exceptions/AppError`:

```typescript
import {
  NotFoundError,
  BadRequestError,
  ValidationError,
  ConflictError,
  UnauthorizedError,
  ForbiddenError
} from '@/app/exceptions/AppError';

// Usage examples
throw new NotFoundError('User not found');           // 404
throw new BadRequestError('Invalid input');         // 400
throw new ValidationError('Email already exists');  // 422
throw new ConflictError('Resource already exists'); // 409
throw new UnauthorizedError('Not authenticated');   // 401
throw new ForbiddenError('Access denied');          // 403
```

### Error Handling Guidelines

- Never throw generic `Error` objects
- Use appropriate exception class for each error condition
- Always provide meaningful error messages
- Error messages should be user-friendly and specific
- Let the global error handler process exceptions
- Use `next(error)` in route handlers to delegate error handling

## Database & Repository Access

Always use TypeORM repositories for database operations:

```typescript
import { AppDataSource } from '@/app/database';

// Initialize repository
private repo = AppDataSource.getMongoRepository(EntityName);

// Common operations
const record = await this.repo.findOne({ where: { _id: new ObjectId(id) } });
const records = await this.repo.find({ where: { is_active: true } });
await this.repo.save(newRecord);
await this.repo.delete({ _id: new ObjectId(id) });
```

### MongoDB ObjectId Handling

- Always validate ObjectId format before querying: `ObjectId.isValid(id)`
- Convert string to ObjectId when querying: `new ObjectId(id)`
- Import from `mongodb` package

// Success responses
res.json({

});

// Error responses (handled by global error middleware)

## Code Style

### General Guidelines

- Use `strict` TypeScript (all strict options enabled)
- Enable `noUnusedLocals` and `noUnusedParameters`
- Use async/await (avoid Promise chains)
- Use arrow functions for small callbacks
- Use optional chaining (`?.`) and nullish coalescing (`??`)
- Prefer `const` over `let`, avoid `var`

### Comments

- Use JSDoc comments for public methods
- Keep comments concise and focused on "why", not "what"
- Remove commented-out code before committing

```typescript
/**
 * Retrieve all active categories with optional filtering
 */
async getAllCategories(filter?: CategoryFilter): Promise<Category[]> {
  // Implementation details
}
```

## Feature-Specific Guidelines

### Guest Features (Unauthenticated)

- Located in `src/app/features/guest/`
- No authentication required
- Read-only operations preferred
- Public data only
- Example: Product browsing, category viewing

### Authenticated Features

- Located in `src/app/features/authenticated/`
- Require Firebase authentication
- Can modify user-specific data
- Example: Customer profile, orders, payments

### System Features

- Located in `src/app/features/system/`
- Internal operations, may require special auth
- Example: Admin operations, system maintenance

## File Extension & Formatting

- Use `.ts` for all TypeScript files
- Use `.json` for configuration files
- Format with consistent indentation (2 spaces)
- Remove trailing whitespace
- Ensure newline at end of file

## Testing & Validation

While full test pattern isn't defined yet, follow:

- Validate all user inputs in routes before passing to services
- Use TypeScript's type system for compile-time safety
- Use `ObjectId.isValid()` for MongoDB ID validation
- Handle edge cases (null, undefined, empty arrays)

## Environment & Configuration

Configuration should be centralized in `src/app/config/index.ts`:

```typescript
export const config = {
  env: process.env.NODE_ENV || 'development',
  port: parseInt(process.env.PORT || '3000', 10),
  apiVersion: process.env.API_VERSION || 'v1',
  // Add other config values
};
```

- Never hardcode values
- Use environment variables for all configuration
- Provide sensible defaults

## Deprecation & Future Patterns

- **Logging**: Use `src/app/logger` for all logging operations
- **Caching**: Implement cache layer in `src/app/caches/`
- **Middleware**: Add all Express middleware to `src/app/middlewares/`
- **Integrations**: External APIs go in `src/app/integrations/`

## Common Mistakes to Avoid

1. ❌ Using relative paths instead of `@/` alias
2. ❌ Mixing business logic between Route and Service
3. ❌ Not validating MongoDB ObjectIds
4. ❌ Returning full entities instead of partial data
5. ❌ Throwing generic `Error` instead of custom `AppError`
6. ❌ Not using try-catch in route handlers
7. ❌ Hardcoding values instead of using config
8. ❌ Mixing snake_case and camelCase in database fields
9. ❌ Creating multiple service instances instead of singleton
10. ❌ Not handling async operations properly with await

## Quick Reference

|-------|---------|----------|---------|
| Route | HTTP endpoints, validation | `{Feature}Route.ts` | Router class with try-catch |

| Exception | Custom errors | `exceptions/AppError.ts` | Extends AppError class |
